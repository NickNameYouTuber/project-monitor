# Стратегия тестирования

## Выбранные уровни
Для первой волны реализации используются два уровня автоматизации:
1. `Unit`-тесты для детерминированной сервисной логики.
2. `API`-тесты на уровне контроллеров для основных бизнес-сценариев.

## Почему выбрана именно эта комбинация
- Она закрывает требование дипломного задания минимум по двум уровням тестирования.
- Она подходит под текущий стек в [newbackend](../../newbackend) без внедрения браузерных инструментов.
- Она рано покрывает наиболее рискованные бизнес-правила: контроль доступа, изоляцию организаций, создание проектов и целостность потока задач.

## Текущие объекты автоматизации
### Unit
- `PipelineYamlParser`
- `CallStatusManager`

### API / controller
- `OrganizationsController`
- `ProjectsController`
- `TasksController`

## Отложенные области
Следующие модули отражены в артефактах планирования, но осознанно перенесены на следующую итерацию, потому что зависят от более тяжелой инфраструктуры или внешних сервисов:
- звонки и WebRTC-сценарии;
- SSO / NIID;
- интеграция с Telegram;
- AI chat;
- межсервисные браузерные E2E-сценарии.
