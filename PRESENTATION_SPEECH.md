## Слайд: Введение, предметная область и проблемная зона

Предметная область — организация работы команд: планирование, задачи, репозитории, мёрдж-реквесты, видеозвонки и чат в одной среде.

Сейчас команды часто используют разные инструменты — трекер, отдельный Git, видеоконференции, мессенджеры. Из-за этого приходится постоянно переключаться между сервисами, дублировать ссылки и настраивать интеграции. Это снижает удобство и усложняет контроль сроков.

---

## Слайд: Цель работы

Цель — разработать веб-приложение, в котором объединены проекты и задачи, репозитории с MR, видеозвонки и чат с AI. В итоге получается единая среда без набора разрозненных подписок.

---

## Слайд: Анализ аналогов (таблица)

Здесь — сравнение с существующими решениями по функциональности, удобству, видеозвонкам и возможности развернуть у себя.

Задача была объединить перечисленные возможности в одном приложении. Jira, GitLab, Trello и аналоги либо закрывают только часть сценариев, либо требуют много интеграций и доплат. В моём решении — полный набор в одной среде плюс развёртывание через Docker.

---

## Слайд: Требования к разработке

Требования заданы через роли: гость — регистрация и вход (логин, Telegram, SSO); пользователь — доступ ко всему функционалу, владелец организации настраивает участников и роли. Технические роли — Runner для CI/CD и Git-клиент. Детали по ролям — на функциональной схеме. Реализация: сервер на Spring Boot (REST, WebSocket), клиент — SPA на React, данные — PostgreSQL, видеозвонки — внешние SFU. Развёртывание — Docker и docker-compose.

## Слайд: Описание используемых платформ и технологий

Backend — Java и Spring Boot, frontend — React и TypeScript; БД — PostgreSQL и Flyway, CI/CD — раннер на Python в Docker. Всё разворачивается через docker-compose в контейнерах. Видеозвонки — отдельные SFU (LiveKit, MediaSoup), медиа идёт напрямую браузер–SFU, backend выдаёт токены.

---

## Слайд: Архитектура программного продукта

Клиент ходит в backend по REST и WebSocket (realtime, логи, уведомления), сессия — в LocalStorage. Backend хранит данные в PostgreSQL, общается с Runner и сервисами видеосвязи. Звонки: браузер по WebRTC к SFU, управление — через backend, медиа — напрямую.

---

## Слайд: Структурная схема

Backend: точка входа, контроллеры, сервисы, репозитории и модели, WebSocket, конфигурация и Git over HTTP. Frontend: SPA с маршрутизацией, страницы, модуль звонков, REST- и WebSocket-клиенты, типы и контексты. Runner и видеосервисы — отдельные образы.
